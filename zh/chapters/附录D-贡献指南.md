# 附录D：贡献指南

欢迎参与《驾驭：一部人类生产力史》的开源协作！本书采用开放协作模式，任何人都可以提出修正、补充内容或翻译贡献。本指南将帮助你快速了解参与方式和规范要求。

---

## 如何 Fork 和提交 PR

### 1. Fork 仓库

点击 GitHub 仓库页面右上角的 **Fork** 按钮，将项目复制到你的个人账户下。

### 2. 克隆到本地

```bash
git clone https://github.com/<你的用户名>/harnessing.git
cd harnessing
```

### 3. 创建分支

请基于 `main` 分支创建新的工作分支，命名规范如下：

```bash
git checkout -b <类型>/<简短描述>
```

分支命名示例：
- `fix/chapter3-typo` — 修正第三章的笔误
- `content/add-steam-engine-section` — 补充蒸汽机相关章节
- `translate/appendix-a-en` — 翻译附录A英文版
- `docs/update-contributing` — 更新贡献指南

### 4. 编辑并提交

```bash
# 编辑文件后
git add .
git commit -m "fix: 修正第三章关于瓦特蒸汽机效率的数据"
```

Commit message 格式：`<类型>: <简短描述>`

类型包括：
| 类型 | 用途 |
|:---|:---|
| `fix` | 修正事实错误、笔误、数据纠错 |
| `content` | 新增或大幅修改正文内容 |
| `translate` | 翻译相关工作 |
| `style` | 格式调整（不改变内容含义） |
| `docs` | 项目文档（非正文）更新 |
| `ref` | 补充或更正参考文献 |

### 5. 推送并创建 Pull Request

```bash
git push origin <你的分支名>
```

然后在 GitHub 上创建 Pull Request，填写：
- **标题**：简明扼要描述变更
- **描述**：说明修改的理由、涉及的章节、参考来源
- **关联 Issue**（如有）：使用 `Closes #123` 语法

### 6. 代码审查

维护者将在 7 个工作日内审查你的 PR。审查重点包括：
- 事实准确性（需标注来源）
- 与全书叙事风格的一致性
- 中英文规范的遵守

## 写作风格指南

为保持全书风格一致，请遵循以下原则：

### 叙事结构

1. **以故事开篇**：每个章节和主要段落应以具体的人物、场景或事件引入，避免直接堆砌抽象概念。
2. **微观到宏观**：从一个具体的技术细节或历史瞬间出发，逐步拉升到文明尺度的影响。
3. **因果链清晰**：明确展示"技术发明 → 能源/效率变化 → 社会组织变革 → 生产力跃迁"的逻辑链。
4. **数据锚定**：关键论点必须有量化数据支撑（功率、效率、产量、人口等），避免纯定性描述。
5. **对比张力**：善用"之前 vs. 之后"、"旧技术 vs. 新技术"的对比来制造叙事张力。

### 语言风格

- **简洁有力**：句子以短句为主（一般不超过 40 字），长句用于必要的逻辑铺展。
- **主动语态优先**：用"瓦特改进了蒸汽机"而非"蒸汽机被瓦特改进"。
- **避免学术黑话**：面向通识读者写作，专业术语首次出现时必须给出简明解释。
- **克制使用形容词**：让数据和事实自己说话，避免"伟大的""划时代的"等空洞修饰。
- **适度使用类比**：将陌生概念映射到读者已知的经验上。

### 引用规范

- 正文中使用脚注格式：`[^1]`，在章节末尾列出完整引用。
- 数据来源必须标注，格式为：`作者, 《书名/论文名》, 出版年, 页码/章节`。
- 网络来源需标注访问日期。

## 翻译规范

本书以中文为主版本，英文为翻译版本。中英文对照遵循以下规范：

### 术语翻译

| 原则 | 说明 | 示例 |
|:---|:---|:---|
| 约定俗成优先 | 已有通行译名的术语直接使用 | Industrial Revolution → 工业革命 |
| 首次出现加注 | 非通用术语首次出现时括号标注原文 | 全要素生产率（Total Factor Productivity, TFP） |
| 专有名词保留 | 品牌、机构缩写等不译 | OECD、GDP |
| 人名规范 | 首次出现全名+原文，后续用姓氏 | 瓦茨拉夫·斯米尔（Vaclav Smil），后续称"斯米尔" |

### 中英文排版

- 中文与英文/数字之间加一个半角空格：`能量密度约为 5000 W/m²`
- 中文标点使用全角：，。！？；：""''（）
- 英文内容（引用原文、代码）使用半角标点
- 数字一律使用阿拉伯数字，四位及以上用逗号分隔：`12,000`
- 年份不加逗号：`1769年`（非 `1,769年`）

### 文件命名

- 中文版路径：`zh/chapters/`
- 英文版路径：`en/chapters/`
- 文件名使用中文（中文版）或英文短横线连接（英文版）

## Issue 标签说明

在 GitHub Issues 中使用以下标签体系：

| 标签 | 颜色 | 用途 |
|:---|:---|:---|
| `bug/事实错误` | 红色 | 正文中的事实性错误（数据、日期、人名等） |
| `bug/笔误` | 浅红 | 错别字、语法错误、格式问题 |
| `enhancement/内容补充` | 蓝色 | 建议补充的内容、遗漏的重要发明或时代 |
| `enhancement/深化` | 浅蓝 | 对现有内容的深化或细化建议 |
| `translation` | 绿色 | 翻译相关（新翻译、译名讨论、翻译质量） |
| `discussion` | 黄色 | 开放性讨论（历史解读、方法论、观点争议） |
| `good first issue` | 紫色 | 适合新贡献者的入门任务 |
| `help wanted` | 橙色 | 需要社区帮助的任务 |
| `wontfix` | 灰色 | 确认不会修改的问题（需说明理由） |
| `duplicate` | 灰色 | 与已有 Issue 重复 |

### 提交 Issue 的模板

创建 Issue 时请选择对应模板并填写完整信息：
- **事实错误报告**：标明章节位置、错误内容、正确内容、来源依据
- **内容建议**：描述建议补充的内容、为什么重要、初步资料来源
- **翻译问题**：标明位置、当前译文、建议译文、理由

## 行为准则

本项目采用 [Contributor Covenant](https://www.contributor-covenant.org/) v2.1 行为准则，核心要求如下：

### 我们的承诺

我们承诺让参与本项目的每个人都能获得无骚扰的体验，无论其年龄、体型、残障状况、种族、性别认同与表达、经验水平、教育背景、社会经济地位、国籍、外貌、信仰或性取向如何。

### 我们的标准

**鼓励的行为：**
- 使用欢迎和包容的语言
- 尊重不同的观点和经验
- 优雅地接受建设性批评
- 关注对社区最有利的事情
- 对其他社区成员表示同理心
- 学术诚信：标注来源，承认不确定性

**不可接受的行为：**
- 使用性暗示的语言或图像，以及不受欢迎的性关注
- 恶意挑衅、侮辱性评论、人身攻击或政治攻击
- 公开或私下骚扰
- 未经明确许可发布他人的私人信息
- 在学术讨论中蓄意歪曲他人观点
- 其他在专业环境中被合理认为不当的行为

### 执行

违反行为准则的行为可通过 [conduct@harnessing-book.org]（示例邮箱）举报。维护团队将在 72 小时内审查并回应所有投诉。

### 处理方式

1. **提醒**：首次轻微违规，私下提醒并要求修正
2. **警告**：重复违规或较严重行为，公开警告
3. **临时封禁**：严重违规，暂停一定时期的社区参与权
4. **永久封禁**：极端情况或屡教不改
